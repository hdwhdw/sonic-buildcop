# Feature Research

**Domain:** Submodule staleness dashboard (GitHub Pages, static site)
**Researched:** 2025-07-18
**Confidence:** MEDIUM — based on domain analysis of comparable dependency-health tooling (Dependabot, Renovate, libyear, deps.dev, David-dm) adapted to the git-submodule niche. No direct competitors exist for cadence-aware submodule dashboards; confidence is MEDIUM because features are synthesized from adjacent domains rather than observed in identical products.

## Feature Landscape

### Table Stakes (Users Expect These)

Features users assume exist. Missing these = dashboard feels broken or useless.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Submodule listing with current pointer** | Core purpose of the dashboard; without it there's nothing to show | LOW | List all 31 sonic-net submodules with their path and current pinned commit SHA |
| **Commits-behind count** | The primary quantitative staleness metric; every dependency tool shows this (Dependabot, Renovate, GitHub compare view) | LOW | `git rev-list pinned..HEAD --count` per submodule |
| **Days-behind / time since pinned commit** | Temporal dimension of staleness; commits-behind alone is misleading if commits are tiny or huge | LOW | Difference between pinned commit date and now |
| **Color-coded status badges (green/yellow/red)** | Visual triage is the whole point; users glance and find problems. Universal in monitoring dashboards | LOW | Three-state: OK / Warning / Critical. Colors must be accessible (not just red/green — use icons too) |
| **Link to submodule repository** | Users need one click to investigate; dead-end dashboards frustrate | LOW | Link to `github.com/sonic-net/{repo}` and to the compare view (pinned..HEAD) |
| **Last-refreshed timestamp** | Users must know if data is stale itself; a dashboard without a timestamp is untrustable | LOW | Show when GitHub Actions last ran successfully |
| **Sort by severity** | With 31 submodules, users need worst-first ordering to prioritize; default sort must be actionable | LOW | Default: sort by staleness severity descending. Also support sort by name, commits behind, days behind |
| **Summary/aggregate view** | Users want a 2-second answer to "how healthy is the project overall?" before diving into details | LOW | Counts: X green, Y yellow, Z red. Maybe a single health percentage |
| **Static GitHub Pages deployment** | Per constraints — must work without a backend; users expect the URL to just work | LOW | Single HTML file or small static site generated by CI |
| **Responsive layout** | Engineers check dashboards on laptops, monitors, and occasionally phones | LOW | CSS grid/flexbox; table that scrolls horizontally on mobile |

### Differentiators (Competitive Advantage)

Features that set this dashboard apart. Not expected (nothing comparable exists for submodules), but these are what make it *useful* rather than just *informational*.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Cadence-aware staleness thresholds** | THE core differentiator. Fixed thresholds are useless when submodule update frequencies range from weekly to yearly. A submodule updated weekly being 14 days behind is alarming; one updated annually being 30 days behind is normal. Auto-compute yellow/red thresholds from each submodule's historical inter-update interval (e.g., 2× median interval = yellow, 4× = red) | HIGH | Requires analyzing git log history of submodule pointer changes in sonic-buildimage. Must handle edge cases: new submodules with no history, submodules that changed cadence, outlier gaps. This is the hardest feature and the most valuable one. |
| **Team ownership grouping** | CODEOWNERS maps submodules to teams (sonic-management, sonic-dataplane, sonic-platform, sonic-kernel, etc.). Grouping by owner lets team leads see "my team's submodules" at a glance. No dependency tool does this for submodules. | MEDIUM | Parse CODEOWNERS file from sonic-buildimage. Some submodules map to individual users, not teams — normalize to teams where possible. ~6 distinct teams visible in CODEOWNERS. |
| **Commit diff summary** | Show the titles/authors of missed commits between pinned and HEAD. Transforms "42 commits behind" from an abstract number into actionable context ("oh, 3 of those are security fixes") | MEDIUM | Fetch commit list from GitHub API (paginated). Must limit display (show first N, link to full compare). API rate limits are a concern if fetching for all 31 submodules. |
| **Historical staleness trend** | Chart showing each submodule's staleness over time (days behind or commits behind per day). Reveals patterns: "this submodule drifts every quarter" or "this one was fixed last month but is drifting again" | HIGH | Requires storing historical snapshots. Each daily CI run would append to a JSON data file committed to the repo. Accumulates over weeks/months. Chart library needed (Chart.js or similar lightweight option). |
| **Update cadence visualization** | Show each submodule's natural update rhythm — a sparkline or bar chart of inter-update intervals. Makes the threshold logic transparent: users can see "this submodule usually updates every 8 days, it's been 20 days" | MEDIUM | Derived from the same historical data as cadence-aware thresholds. Visualization is the added value beyond just using the data for threshold computation. |
| **Staleness heatmap / distribution** | A single visual showing all 31 submodules as a grid/matrix colored by staleness. Immediate gestalt of project health. Similar to GitHub's contribution heatmap but for submodule health. | LOW | Simple grid of colored cells. Low complexity if you already have the staleness data computed. |
| **Compare view link** | Direct link to `github.com/sonic-net/{repo}/compare/{pinned_sha}...HEAD` for each submodule. One click to see the full diff of what's been missed. | LOW | URL construction only — no API calls needed. Extremely high value-to-effort ratio. |
| **JSON data endpoint** | Expose the computed staleness data as a JSON file alongside the HTML dashboard. Enables downstream consumers: scripts, Slack bots, other dashboards, CI gates. | LOW | Already generating JSON as intermediate data; just make it publicly accessible at a stable URL. |

### Anti-Features (Commonly Requested, Often Problematic)

Features that seem good but create problems in this context.

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| **Auto-update submodule pointers** | "If you know it's stale, just fix it" | Submodule updates can break builds; they need testing. The automerge workflow (`automerge_scan.yml`) exists but is disabled for good reason. Observability ≠ automation. Conflating them destroys trust in the dashboard. | Dashboard links to the compare view; humans decide when to update. Flag submodules that have open update PRs from the bot. |
| **GitHub Issues / Slack / email alerting** | "Notify owners when things go stale" | Adds significant complexity (GitHub API write permissions, secrets management, notification fatigue). Explicitly deferred to v2 in PROJECT.md. Alerting before the staleness model is validated = noise. | v1 is a pull-based dashboard. Validate thresholds are meaningful first. Add push notifications in v2 after confidence in the model. |
| **Tracking non-sonic-net submodules** | "Why only 31? Track all 49" | External submodules (p4lang, Marvell, Nokia, Mellanox, Arista) have different ownership models, update expectations, and no CODEOWNERS mapping. Mixing them dilutes the signal. | Explicitly out of scope per PROJECT.md. Could add as a separate "external dependencies" section in v2 with different thresholds. |
| **Real-time / webhook-driven updates** | "Why wait for daily cron?" | GitHub Pages is static; real-time requires a server. Daily cadence is sufficient for submodule drift (you're measuring weeks/months of staleness, not minutes). Webhook infrastructure adds hosting cost and complexity. | Daily GitHub Actions cron is the right cadence. Add manual "refresh now" workflow_dispatch trigger for on-demand updates. |
| **Database backend** | "Store historical data in PostgreSQL/SQLite" | Violates GitHub Pages constraint. Adds infrastructure, cost, and maintenance. For 31 submodules × daily snapshots, data volume is trivial. | Commit a JSON file to the repo on each CI run. Git itself becomes the database. After 1 year: 365 × ~5KB = ~1.8MB. Negligible. |
| **User accounts / authentication** | "Let team leads customize their view" | Public dashboard for a public repo. Auth adds massive complexity for zero value. Customization can be done client-side. | URL query parameters for filtering (e.g., `?team=sonic-management`). Client-side localStorage for preferences. |
| **Complex configuration UI** | "Let users change thresholds in the dashboard" | Thresholds are auto-computed; manual overrides should be rare and code-reviewed. A UI for config changes bypasses version control. | YAML config file in the repo for manual threshold overrides. Changed via PR, reviewed by team. |
| **PR creation for stale submodules** | "Dashboard should create update PRs" | Requires write permissions to sonic-buildimage (which lives in a different org). The `mssonicbld` bot already creates these PRs. Duplicating that work creates conflicts. | Show status of existing bot PRs alongside staleness data. "This submodule is stale, but PR #1234 is open to update it." |

### Dependency Notes

- **Color-coded badges require cadence-aware thresholds:** You cannot assign green/yellow/red without knowing what "stale" means for each submodule. This is the critical dependency — the threshold logic MUST exist before badges are meaningful. A temporary fallback could use fixed thresholds (e.g., 30/90 days) while cadence analysis is being built.
- **Historical trends require accumulated snapshots:** Trend charts are useless on day 1. The data collection must run daily for weeks before trends become visible. Start collecting immediately even if the visualization comes later.
- **Team grouping requires CODEOWNERS parsing:** The CODEOWNERS file has a specific format (path patterns → GitHub usernames/teams). Not all submodules have clean team mappings — some map to individual users. Need a normalization layer.
- **Commit diff summary has API rate limit risk:** Fetching commit lists for 31 submodules via GitHub API could hit rate limits (5000/hour for authenticated, 60/hour for unauthenticated). Must use authenticated requests and potentially cache results.

## MVP Definition

### Launch With (v1)

Minimum viable product — validate that the staleness model is useful.

- [ ] **Submodule listing with commits-behind and days-behind** — Core data that answers "how stale is each submodule?"
- [ ] **Cadence-aware threshold computation** — The whole reason this project exists; without it, the dashboard is just a worse version of `git submodule status`
- [ ] **Color-coded status badges** — Visual triage; the 2-second glance that makes the dashboard useful
- [ ] **Summary counts** — "3 critical, 5 warning, 23 healthy" header
- [ ] **Compare view links** — One click from "stale" to "here's exactly what changed"
- [ ] **Sort by severity (default)** — Worst offenders first
- [ ] **Last-refreshed timestamp** — Trust signal
- [ ] **GitHub Actions daily cron** — Automated refresh
- [ ] **GitHub Pages deployment** — Accessible at a stable URL

### Add After Validation (v1.x)

Features to add once the staleness model is validated and the dashboard is in daily use.

- [ ] **Team ownership grouping** — Add when >1 team is regularly checking the dashboard; trigger: someone asks "show me just my team's submodules"
- [ ] **Commit diff summary** — Add when users report "I see it's 42 commits behind but I don't know if I should care"; requires GitHub API token in CI
- [ ] **JSON data endpoint** — Add when someone wants to consume the data programmatically; near-zero effort since the data pipeline already produces JSON
- [ ] **Staleness heatmap** — Add when the grid has been running long enough to validate the color logic works
- [ ] **Historical JSON snapshots** — Start collecting from day 1 (just append to a file), but defer the visualization. Low cost to collect, high cost to visualize. Collecting early means you have months of data when you're ready to chart.
- [ ] **Update cadence sparklines** — Add alongside or after threshold validation; makes the "why is this yellow?" question self-answering

### Future Consideration (v2+)

Features to defer until the dashboard is proven useful.

- [ ] **Historical trend charts** — Requires accumulated data + charting library; defer until 2-3 months of snapshots exist
- [ ] **GitHub Issues alerting** — Deferred per PROJECT.md; requires validated thresholds + write API access
- [ ] **Slack/webhook notifications** — Same as above; push notifications only make sense after pull-based model is trusted
- [ ] **Bot PR status integration** — Show if `mssonicbld` has an open PR for each stale submodule; requires searching PRs in sonic-buildimage
- [ ] **External submodule tracking** — If the dashboard proves valuable, extend to non-sonic-net submodules with separate thresholds/expectations
- [ ] **CI gate integration** — Block merges if staleness exceeds critical thresholds; extremely high-impact, needs very confident thresholds

## Competitor / Comparable Tool Analysis

No direct competitor exists for cadence-aware git submodule staleness dashboards. The closest analogues are dependency freshness tools designed for package managers:

| Feature | Dependabot (GitHub) | Renovate | libyear | deps.dev | Our Approach |
|---------|---------------------|----------|---------|----------|--------------|
| **Staleness detection** | Version behind latest | Version behind latest | Time behind latest (in "libyears") | Version + advisory data | Commits behind + days behind + cadence-relative severity |
| **Threshold model** | Binary: outdated or not | Configurable schedules | Single "libyear" metric | N/A (informational) | Auto-computed from historical update cadence per submodule |
| **Visualization** | PR list, security tab | PR list, dashboard (app) | CLI table output | Web dashboard | Static HTML dashboard on GitHub Pages |
| **Ownership mapping** | N/A | N/A (can assign reviewers) | N/A | N/A | CODEOWNERS-based team grouping |
| **Automation** | Creates PRs automatically | Creates PRs automatically | Read-only | Read-only | Read-only (observability only) |
| **Git submodule support** | ❌ Packages only | ❌ Packages only | ❌ Packages only | ❌ Packages only | ✅ Purpose-built for submodules |
| **Historical trends** | N/A | N/A | N/A | Historical advisories | Daily snapshots accumulated in git |
| **Cost** | Free (GitHub native) | Free (open source) | Free (open source) | Free (Google) | Free (GitHub Actions + Pages) |

**Key insight:** Existing tools don't solve this problem. They track *package* versions, not *git submodule pointers*. And none of them compute staleness relative to a component's natural update rhythm. This dashboard fills a genuine gap.
